# Elementify Stack Profile Model

## Purpose

This document defines the canonical stack-profile model for Elementify Free and Advanced.

It translates the scenario and intent model into curated supported starting points across:

- WordPress
- theme baseline
- Elementor
- supported addon profiles
- optional premium upgrades

It should prevent two opposite failures:

- underpowered baseline recommendations
- plugin inflation disguised as guidance

## Role in the Product

Stack profiles are not random bundles.

They are:

- curated baseline recommendations
- scenario-fit operating starting points
- support-aware upgrade anchors

They should always remain subordinate to:

- user intent
- current site state
- scenario depth
- support confidence

## Stack Profile Layers

Each recommendation should distinguish clearly between three layers:

### 1. Baseline stack

The smallest sensible supported working stack.

Example components:

- WordPress
- Hello Theme or a compatible baseline
- Elementor Free

### 2. Addon profile

An optional curated bundle that extends the baseline when the scenario materially benefits.

Examples:

- free utility widgets
- content-marketing helpers
- conversion-oriented free enhancements

### 3. Upgrade path

A justified next step when the scenario needs more workflow depth, more reliable delivery, or more structural power.

Examples:

- Elementor Pro
- Elementify Advanced
- selected supported pro addon profile

## Free Stack Profiles

## FSP-1 — Minimal guided baseline

Best for:

- idea-only starts
- simple service sites
- low-risk first-site builds
- users who want the clearest possible path

Profile:

- WordPress
- Hello Theme
- Elementor Free
- no addon profile by default

Why it exists:

- this is the least noisy, most teachable starting point
- it keeps Free honest and approachable

## FSP-2 — Baseline plus curated free addon profile

Best for:

- users who genuinely need a few richer free building blocks
- scenarios where the baseline alone would create unnecessary friction

Profile:

- WordPress
- Hello Theme
- Elementor Free
- one supported free addon profile when justified

Rules:

- recommend only one curated addon profile by default
- do not stack multiple addon profiles casually
- explain why the addon profile is helpful

## FSP-3 — Brownfield staged transition baseline

Best for:

- existing WordPress sites without Elementor
- classic-editor or manual-theme setups
- low-risk migration starts

Profile:

- current WordPress site retained
- current theme assessed
- Elementor Free introduced carefully
- Hello Theme or compatible baseline considered, not assumed

Rules:

- preserve continuity where reasonable
- do not force immediate full rebuilds
- treat theme replacement as a decision, not a default

## Advanced Stack Profiles

## ASP-1 — Advanced productivity baseline

Best for:

- users already operating in Elementor and wanting deeper workflow reduction
- teams or freelancers doing repeated delivery work

Profile:

- WordPress
- Hello Theme or approved compatible baseline
- Elementor Free or Pro depending scenario fit
- Elementify Advanced
- no extra pro addon profile by default

Why it exists:

- Advanced should already feel meaningfully better before plugin stacking begins

## ASP-2 — Advanced plus curated premium addon profile

Best for:

- deeper creation or rollout scenarios
- stronger component needs
- cases where a premium addon materially reduces friction

Profile:

- Advanced productivity baseline
- one curated supported pro addon profile when justified

Rules:

- the addon profile must solve a clear scenario problem
- do not recommend premium addons as generic default upsell
- support confidence must be explicit

## ASP-3 — Deep relaunch / migration profile

Best for:

- deep relaunch
- deep migration
- larger structural or workflow-heavy delivery paths

Profile:

- current site context retained
- stronger diagnostic layer
- stronger rebuild and critique support
- Elementor Pro, Elementify Advanced, and curated supported addons only when scenario-fit is clear

## Profile Selection Rules

Profile selection should consider, in order:

1. user intent
2. current stack readiness
3. scope depth
4. operational constraints
5. support confidence
6. only then commercial fit

## Theme Baseline Rule

Hello Theme should be treated as:

- the current default baseline
- the most teachable safe starting point
- not the only possible long-term foundation

The system should remain open to:

- compatible retained themes in brownfield scenarios
- a later Elementify Theme or Elementify Theme System direction

But that future should not distort current recommendation honesty.

## Addon Profile Rule

Addon profiles should be:

- curated
- named
- documented
- support-aware

Addon profiles should not be:

- an open plugin marketplace inside the recommendation layer
- a hidden upsell lever
- a substitute for poor baseline choices

## Recommendation Output Contract

A stack recommendation should always be able to state:

- recommended baseline stack
- optional addon profile
- why the addon profile is or is not recommended
- whether Free is enough
- whether a deeper paid path is justified

## Binary Success Criteria

- Free and Advanced stack profiles are described as curated profiles, not arbitrary plugin bundles.
- The profile model distinguishes baseline, addon profile, and upgrade path.
- Hello Theme is positioned as the current default baseline without overclosing future theme evolution.
- The model supports both new-site and existing-site bootstrap scenarios.
